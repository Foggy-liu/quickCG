# Quick Cognition v2（模块化版）

> **单点认知，深度建图。** 本版本从 `quick-cognition/` 拆分而来，按事物类型分发到不同模式。

---

## 核心思路

quick-cognition v2 不是"一个 skill 处理所有事物"，而是"一个路由器 + 多个专属模式"。

**价值主张**：对**任何**事物提供有结构的认知入口——但不同事物用不同结构。

---

## 第一步：识别事物类型（强制）

按顺序问以下问题，**第一个匹配的模式**即为所选模式：

### 1. 它有歧义吗？
- 例子："苹果"（水果/公司）、"小米"（粮食/公司）
- → **询问用户澄清**，根据澄清结果重判

### 2. 它是人 / 公司 / 组织吗？
- 例子：巴菲特、乔布斯、王阳明、华为、特斯拉、OpenAI
- → **人物/公司模式**：加载 [`modes/person.md`](modes/person.md)

### 3. 它是事件 / 现象吗？
- 例子：2008 金融危机、双 11、内卷、996、鸡娃
- **判定标准**：
  - 有明确起止时间 **或** 持续性的社会现象
  - 涉及多方参与者
  - 不是"永恒对象"——会"开始 → 演变 → 结束（或持续）"
- → **事件/现象模式**：加载 [`modes/event.md`](modes/event.md)

### 4. 它是抽象概念吗？
- **判定标准**：
  - 无物理形态
  - 无组成结构
  - 评价因人而异
- 例子：自由、爱、熵、正义、美、真、善
- → **抽象概念模式**（essay 模式）：加载 [`modes/abstract.md`](modes/abstract.md)
- ⏳ **当前状态**：abstract.md 仍在迭代中（roadmap），目前对抽象概念的认知质量有限

### 5. 它是具体技术 / 概念 / 方法 / 工具 / 算法 / 协议 / 框架 / 模型吗？
- 例子：agent、RAG、Transformer、ReAct、MCP、ChromaDB、Function Calling、Self-Correction、LangChain、bge-large
- → **概念/技术模式**：加载 [`modes/concept.md`](modes/concept.md) — **默认模式**（✅ **目前唯一完整可用的模式**）

> ⚠️ **半成品说明**：本 skill 目前仅 concept 模式经过真实概念验证（RAG/Docker）。event/person/abstract 模式文件存在但规则未跟上 v2.4 双层卡机制，使用时可能产生质量不稳定。

---

## 第二步：执行对应模式

每个 mode 文件是**自包含**的，包含：
- 适用对象与边界
- 强制结构（板块 + 字数 + 必标注项）
- 强制规则（"不遵守则重新生成"）
- 必启扩展维度
- 禁用清单
- 交付前质检

**AI 必须严格按 mode 文件执行**——不允许"自由发挥"或"选择性执行"。

---

## 第二步.5：双层预收集卡（写报告前必填，**先于任何写作**）

**为什么需要**：v2.3 的单层卡虽然根治了事实错误，但 AI 写得过保守（OpenClaw 报告 6500 < 8000 字）。**v2.4 改为双层**：A 段"事实地基"严格（必带 [N]）+ B 段"推断上层"宽松（基于事实、标"推断"、无需来源）。**事实和灵感可以并存**。

**写报告前，AI 必须先输出下面这张"双层预收集卡"**：

```markdown
### A. 事实地基（必须可追溯，标 [N]）

| # | 类型 | 拟用事实 | 来源 [N] | 可信度 |
|---|------|---------|---------|--------|
| 1 | 数字 | 28.4 BLEU | [1] | [✓ 可靠] |
| 2 | 日期 | 2017-06-12 论文提交 | [1] | [✓ 可靠] |
| 3 | 架构 | AlphaFold 2 用 Evoformer | [?] | [? 未知] |

### B. 推断上层（基于 A 段，标"推断"无需 [N]）

| # | 拟写推断 | 引用的事实（来自 A 段）|
|---|---------|---------------------|
| 1 | "Transformer 是范式重构而非更好模型" | 引用 A-#1 (28.4 BLEU) + A-#2 (2017 论文) |
| 2 | "个人项目 5 年周期是正常的" | 引用 A-#X (573 commits, 5 年) |
```

### 报告中的标注规则

- **事实层**（来自 A 段）→ 紧跟 `[✓ 可靠][N]` 或 `[? 未知]`
- **推断层**（来自 B 段）→ 紧跟 `[△ 推断]`，**不要** `[✓ 可靠]`

**示例**：

```html
2017 年 8 位作者提出 Transformer <span class="tag tag-reliable">[✓ 可靠][1]</span>
个人项目 5 年周期是正常的 <span class="tag tag-inference">[△ 推断]</span> 引用事实 #X
```

### 关键规则

- [ ] A 段**所有**具体数字/日期/人名/作者数/架构归属**必须**有 `[N]` 来源
- [ ] A 段没有的 → 报告里**不出现**该具体事实（可以写"基于以下事实的推断"）
- [ ] A 段找不到来源的 → 标 `[? 未知]`，**不**编造
- [ ] B 段**每条推断**必须引用 A 段的具体事实编号
- [ ] B 段**严禁**直接断言具体数字/日期/人名——那是 A 段的事
- [ ] B 段占总字数 ≤ 50%（事实主导）

### 这是什么原理

**事实地基 = 数据库 schema**（严格、不允许绕过）
**推断上层 = 查询结果**（基于 schema、可以发挥）
**报告 = 应用层**（只能用 schema 定义的数据 + 用 schema 数据的合理查询）

科学论文的"Results / Discussion"结构同理：Results 是事实，Discussion 是推断——读者能区分。

### 利基话题来源豁免

**判定标准**：执行标准搜索步骤（见 concept.md）后，**如果来源数 < 3**：
- 自动标"**利基/小众概念**"
- 来源要求降为 ≥ 2
- 事实地基里**必须**说明"为何来源少"（如"该概念 2017 年后无新活动，公开来源有限"）

**不**需要预设"什么是小众"——**搜索结果 < 3 就是小众的客观证据**。

## 第三步：交付前强制关卡（不通过则重新生成）

报告生成后，**必须**在报告末尾的"质检报告"小节中回答以下 5 个问题。**任一为"否"则必须修改后重新提交**：

- [ ] Q1: 联网搜索次数 ≥ 5？
- [ ] Q2: 至少 1 处 `[? 未知]` 标签？
- [ ] Q3: AI 味词 grep 命中 ≤ 3？
- [ ] Q4: 来源台账中 ≥ 3 条可追溯 URL？
- [ ] Q5: 报告末尾有"质检报告"小节？

**这 5 个问题是不允许"跳过"的硬规则**——这是从 `quick-cognition/` 旧版的失败中提取的核心教训（参考 `quick-cognition/reports/agent-v2-quality-review.md`）。

---

## 第三步.5：事实核查

**报告里所有具体数字、日期、人名、作者数、架构归属**必须**可追溯到来源台账**——文中紧跟 `[N]` 标注。

不确定的 → 标 `[? 未知]`，**不要**标 `[△ 推测]`。

**Transformer 报告的 3 个错误案例**：

| 错 | 应该怎么写 |
|---|----------|
| "BERT 11 位作者" | "4 位作者 [✓ 可靠][3]" 或 "作者若干 [? 未知]" |
| "GPT-2 于 2018 年 11 月发布" | "GPT-2 论文 2019 年 2 月发布 [✓ 可靠]" |
| "AlphaFold 2 用 Transformer" | "AlphaFold 2 用 Evoformer [? 未知]" |

详细规则见 `modes/concept.md` 末尾"事实核查"区块。

## 第四步：保存与展示（强制动作）

### 4.1 保存

- **路径**：`reports/{事物名}-v{N}.html`（N 是版本号，从 1 开始）
- **同名规则**：若 `{事物名}-v{N}.html` 已存在，N 自动 +1（不允许覆盖）
- **附带证据**：建议同时保存一份源 markdown（`{事物名}-v{N}.md`）便于以后审计和修订——非强制但强烈推荐

### 4.2 浏览器自动打开（强制）

**HTML 报告生成后，必须立即用浏览器打开给用户查看**——**不要等用户说"打开"**。

**核心原则**：**直接调用浏览器 exe 的绝对路径**，完全绕开 Windows 的 `.html` 文件关联和 `file://` 协议 hook（这两个机制都可能被 IDE 如 HBuilder/VS Code 劫持）。

#### Windows（首选：检测 + 直接 exe 调用）

```bash
# 按优先级检测可用浏览器，第一个找到的就用
for browser in \
  "C:/Program Files/Google/Chrome/Application/chrome.exe" \
  "C:/Program Files (x86)/Microsoft/Edge/Application/msedge.exe" \
  "C:/Program Files/Mozilla Firefox/firefox.exe"; do
  if [ -f "$browser" ]; then
    "$browser" "file:///{绝对路径}"
    exit 0
  fi
done

# 都未找到：提示用户手动打开
echo "未检测到 Chrome/Edge/Firefox，请手动打开报告：{绝对路径}"
```

**为什么必须直接调用 exe 绝对路径**：
- `start "" "path"` 会调用 `.html` 系统关联（可能被劫持到 HBuilder/VS Code/记事本）
- `rundll32 url.dll,FileProtocolHandler` 会被 `file://` 协议 hook 劫持
- **只有直接调用浏览器 exe 才能 100% 绕开所有 Windows 关联机制**

**为什么不固定用某个浏览器**：
- 不同用户机器装的浏览器不同
- 检测链按"用户最常用 → 系统预装"优先级排序
- Chrome 优先（开发者首选），Edge 次之（Win 预装），Firefox 兜底

#### Mac

```bash
# 优先级：Chrome > Safari > 默认浏览器
if [ -f "/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" ]; then
  "/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" "file:///{绝对路径}"
else
  open "{绝对路径}"  # 用默认浏览器
fi
```

#### Linux

```bash
# 优先级：Chrome > Chromium > Firefox > xdg-open
for browser in \
  "/usr/bin/google-chrome" \
  "/usr/bin/chromium-browser" \
  "/usr/bin/firefox"; do
  if [ -f "$browser" ]; then
    "$browser" "file:///{绝对路径}"
    exit 0
  fi
done
xdg-open "{绝对路径}"  # 兜底
```

#### chrome-devtools MCP（如可用）

```
navigate_page → file:///{绝对路径} → take_screenshot
```

**不可用时**：在终端输出"已生成报告：{路径}"，明确告知用户打开方式。

**为什么强制**：
- 报告的视觉体验是核心交付物之一（模板、配色、ASCII 图表）
- 用户无法通过"读完文字"判断报告质量
- 不打开 = 没有完成交付

**v2.4.2 教训**：
- v2.4.1 的 `rundll32 url.dll,FileProtocolHandler` 在某些机器上仍被 IDE 劫持（HBuilder 注册了 `file://` 协议 hook）
- `start "" "path"` 触发 `.html` 关联被劫持
- **唯一可靠方案：直接调用浏览器 exe 的绝对路径**——完全绕开 Windows 关联机制

### 4.3 视觉自检（打开后立即做）

打开后**必须**自检 3 项，**任何 1 项失败必须修复后重新生成**：

- [ ] **首屏可见 TL;DR 摘要**——用户打开 1 秒内能看到核心结论
- [ ] **ASCII 架构图清晰可读**——没有出现方块乱码、字符重叠
- [ ] **4 个色块（历史/结构/功能/本质）层级分明**——颜色对比明显、板块标题清晰

### 4.4 导出（可选）

- 用户可要求导出为纯文本笔记（去掉 HTML 标签）
- 导出格式：`reports/{事物名}-v{N}.txt`

### 4.5 文档说明（v2.4.3 强制）

**为什么需要**：AI 生成的报告可能包含**幻觉事实**（如编造版本号、误判公司历史）。即使用了 `[✓ 可靠][N]` 标注，也**不能 100% 保证事实准确性**——标注只能反映"AI 自认为有来源"，不能替代真实核实。

**强制要求**：报告**必须**在"信息来源"和"质检报告"之间显示以下声明（模板已内置，AI 无需手写）：

> 📝 文档说明：本报告由 AI 辅助生成，**仅供参考**。涉及具体事实、数字、版本号、日期等信息，请以原始来源（见上方"信息来源"）为准。

**设计意图**：
- **不试图隐藏 AI 生成的事实**——主动告知用户"这是 AI 写的"
- **把判断权交回用户**——鼓励读者对关键事实回到原始来源核实
- **降低信任风险**——避免用户因"看起来很专业"而误信幻觉内容

**为什么放在来源后、质检前**：
- 来源（"我参考了什么"） → 免责声明（"AI 写的，仅供参考"） → 质检（"我自评的过程合规性"）
- 三层信息梯度递进：证据 → 限制声明 → 自评

## 文件结构

```
quick-cognition-v2/
├── SKILL.md          ← 本文件（路由器）
├── modes/
│   ├── concept.md    ← 概念/技术模式（默认）
│   ├── event.md      ← 事件/现象模式
│   ├── person.md     ← 人物/公司模式
│   └── abstract.md   ← 抽象概念引导（轻量）
├── templates/
│   └── report.html   ← 共享 HTML 模板
└── reports/          ← 生成的报告（v2 版）
```

## 触发词

- 「帮我认知一下 XX」（XX 必须是具体技术/概念）
- 「了解一下 XX」
- 「深度研究 XX」（如果 XX 是技术/概念）
- 「认识一下这个东西」（指代一个具体技术）
- 「快速认知 XX」

**如果用户输入的 XX 触发了歧义/事件/抽象/人物**——按第一步的路由规则处理。
