# 概念/技术模式（默认模式）

> **适用对象**：具体技术、概念、方法、工具、算法、协议、框架、模型、架构模式。
>
> **典型例子**：agent、RAG、Transformer、ReAct、MCP、ChromaDB、Function Calling、Self-Correction、LangChain、bge-large、Vector Search、HNSW、BM25、OpenAPI、JSON Schema

---

## 强制结构

| 板块 | 必启 | 字数下限 | 必标注可信度 |
|------|------|---------|------------|
| TL;DR | ✓ 必启 | 30-50 字 | - |
| 历史维度 | △ 默认启 | 2000 字 | 每段 |
| 结构维度 | △ 默认启 | 1500 字 | 每段 |
| 功能维度 | △ 默认启 | 1500 字 | 每段 |
| 本质维度 | ✓ 必启 | 1000 字 | 每段 |
| 横纵交汇洞察 | ✓ 必启 | 800 字 | - |
| 回到本质 | ✓ 必启 | 500 字 | - |
| **总计** | - | **8000 字+** | - |

> **△** = 默认启用但可豁免（见下方"维度自适应规则"）
> **✓** = 永远强制启用

## 维度自适应规则（核心改进 v2.1）

**4 个核心维度不是铁板一块——根据事物性质可豁免 1 个，最多豁免 1 个**。

### 为什么需要自适应

不是所有概念都"完美对齐"4 维结构：

| 情况 | 例子 | 表现 |
|------|------|------|
| **缺乏清晰时间线** | 数学定理（如欧拉公式 `e^(iπ)+1=0`）、纯算法（如快速排序） | "历史"只能写 1-2 段（起源 + 标准化），硬写 2000 字会注水 |
| **结构即本质** | 简单的数据结构（如 `Stack`、`Queue`）、极简协议（如 HTTP GET） | "结构"和"本质"高度重合，强行拆开是凑字数 |
| **没有"功能"可言** | 纯理论框架（如 `λ 演算`）、纯数学对象（如 `范畴论`） | "功能"只能写"在哪些领域被应用"，本质是"应用"而非"功能" |
| **没有"结构"可言** | 单一概念（如 `熵` 在物理学语境、`一致性` 在分布式语境） | "结构"会变成空话，需要诚实承认 |

### 自适应规则（强制）

- [ ] **AI 必须在生成报告前，先判断是否豁免某个维度**（最多豁免 1 个）
- [ ] **判断结果必须显式记录在"质检报告"中**（"维度决策：历史（启用）/ 结构（启用）/ 功能（启用）/ 本质（必启）"）
- [ ] **如果豁免某个维度，必须给出 ≥ 2 条具体理由**（不能说"不太适用"——必须指出具体原因）
- [ ] **豁免维度后，总字数要求不变**——剩余维度的字数下限**按比例上调**（例：豁免"功能"后，3 个维度各上调 33%）
- [ ] **横纵交汇洞察必须仍然讨论"被豁免的维度"**——用 1 段话讲"为什么这个事物不适合这个维度"（这本身是有价值的洞察）

### 豁免决策清单（写报告前自问）

**是否豁免"历史"维度？**
- [ ] 该事物是数学定理/公式/纯算法？
- [ ] 该事物没有清晰的"出现 → 演化 → 标准化"过程？
- [ ] 公开资料中关于起源的描述 ≤ 3 段？
→ 满足 ≥ 2 项可豁免

**是否豁免"结构"维度？**
- [ ] 该事物是单一概念/单一动作/单一参数？
- [ ] 没有"组件"和"组件关系"可言？
- [ ] "结构"和"本质"会高度重复？
→ 满足 ≥ 2 项可豁免

**是否豁免"功能"维度？**
- [ ] 该事物是纯理论/纯框架/纯方法论？
- [ ] 它的价值在"如何思考"而非"能做什么"？
- [ ] 应用案例都是"借用"而非"原生功能"？
→ 满足 ≥ 2 项可豁免

**是否豁免"本质"维度？**
- ❌ **永远不豁免**——本质是 quick-cognition 的核心交付物

### 字数上调公式

```
原总字数 = 8000
原 4 维度字数 = 历史(2000) + 结构(1500) + 功能(1500) + 本质(1000) = 6000
剩余字数 = 6000（不变）
剩余维度数 = 3
每维度新字数 = 6000 / 3 = 2000
```

**简化为**：豁免 1 个维度后，其余 3 个维度字数**平均 2000 字**（不必精确分配，可上下浮动 30%）。

### 与扩展维度的关系

**4 个核心维度的"是否豁免"和 4 个扩展维度的"是否启用"是两套独立决策**：

| 决策类型 | 适用范围 | 决策点 |
|---------|---------|--------|
| 核心维度豁免 | 历史/结构/功能（本质永远必启） | 写报告前 |
| 扩展维度启用 | 优缺点/常见误解/类比解释/同类对比 | 写报告过程中 |

**强制**：两套决策都必须在"质检报告"中显式记录理由。

## 四维边界定义（不允许越界）

| 维度 | 唯一回答 | 禁止出现 |
|------|---------|---------|
| **历史** | "它**怎么走到今天**？" | 不写"能做什么"（属功能）、"是什么"（属本质） |
| **结构** | "它由**哪些部分**组成、**怎么组织**？" | 不写历史时间线、不写应用场景 |
| **功能** | "它能**做什么**、**做不到什么**？" | 不写"是什么"（属本质）、不写组件关系（属结构） |
| **本质** | "它**最不可撼动**的核心是什么？" | 不罗列能力、不讲历史 |

**冲突仲裁原则**：如果一段内容同时属于多个维度，**优先放在问题最匹配的那一个**，其他维度用 1 句话 callback，**不**整段重复。

## 字数计算规则

**"字数"仅指分析性散文**，**不**包括：
- HTML 标签和 CSS
- 表格内容（表格是结构化展示，不计入正文）
- 代码块 / 引用块 / 卡片内容
- **信息来源列表**（包括来源台账、URL、访问时间）
- 可信度标签（`[✓ 可靠]` `[△ 推测]` `[? 未知]`）

**只计入**：`<p>`、`<h4>` 标题、分析性叙述、列表项中的论述文字。

**目的**：确保"凑字数"和"加来源"是两条独立路径，不会互相污染。

## 关键概念完整性（写报告**前**必填，**先于任何写作**）

**为什么需要**：Transformer 报告漏讲了 Cross-Attention / Causal Mask / 训练细节 / 现代变体——因为 SKILL 只在"事后"列清单，AI 写报告时**容易挑容易讲的概念**。**根治办法是"事前提取+占位符"**：先勾选类型 → 在草稿里**先留占位符** → 写报告时**占位符必须填满**。

### 写报告前必做（按顺序）

**Step 1：勾选适用类型**

```markdown
- [ ] A. Seq2Seq / 编码器-解码器（如 Transformer、T5、BART）
- [ ] B. 自回归生成（如 GPT、LLaMA、Decoder-only）
- [ ] C. 可训练 ML 模型（含训练过程的所有模型）
- [ ] D. 成熟架构（2+ 年历史的，如 Transformer、ResNet）
- [ ] E. 算法 / 方法论（无架构，如快排、MapReduce、CAP）
```

**Step 2：在草稿里**先**留占位符**（每勾一个类型，留一个占位符）

```markdown
<!-- 类型 A 占位符（若勾选） -->
<!-- 必填：Cross-Attention 工作原理 + 为什么需要 -->

<!-- 类型 B 占位符（若勾选） -->
<!-- 必填：Causal Mask 工作原理 + 为什么需要 -->

<!-- 类型 C 占位符（若勾选） -->
<!-- 必填：训练数据/优化器/学习率/正则化/算力 -->

<!-- 类型 D 占位符（若勾选） -->
<!-- 必填：现代变体 + 当前争议 -->

<!-- 类型 E 占位符（若勾选） -->
<!-- 必填：输入输出 + 核心步骤 + 复杂度分析 -->
```

**Step 3：写报告**——每个占位符**必须**填上对应内容。

**这是根治"漏讲 Cross-Attention"的唯一办法**——占位符没填 = 没写完。

### 5 个类型的必讲内容速查

| 类型 | 必讲 |
|------|------|
| **A** Seq2Seq | Cross-Attention 怎么工作 + 为什么需要 |
| **B** 自回归 | Causal Mask 怎么工作 + 为什么需要 |
| **C** 可训练模型 | 训练数据 + 优化器 + 学习率 + 正则化 + 算力 |
| **D** 成熟架构 | 现代变体 + 当前争议 |
| **E** 算法 | 输入输出 + 核心步骤 + 复杂度 |

## 时效性（写报告**前**必搜，**先于任何写作**）

**为什么需要**：Transformer 报告写到了 2026 年但**完全没提 2024-2026 进展**（SSM/Mamba 等）——AI 写报告时**容易忽略近况**。**根治办法是"事前提取"**：先搜近 2 年进展 → 至少 1 个进来源台账 → 写报告时**必须用上**。

### 写报告前必做（按顺序）

**Step 1：搜近 2 年进展**（联网搜，**先于任何写作**）

```markdown
- [ ] 搜 `"{概念} 2024 进展"` → 找到 ___ 个新进展
- [ ] 搜 `"{概念} 2025 进展"` → 找到 ___ 个新进展
- [ ] 搜 `"{概念} 2026 进展"` → 找到 ___ 个新进展
```

**Step 2：把 ≥ 1 个近 2 年进展写进来源台账**

```markdown
| [N] | 《来源名》 | URL | 访问时间 | 关键事实 |
```

**Step 3：在报告草稿里**留占位符**

```markdown
<!-- 时效性占位符（必填 ≥ 1 个近 2 年进展） -->
<!-- 位置建议：历史维度末尾 或 横纵交汇 -->
```

**Step 4：写报告**——占位符**必须**填上具体进展。

**这是根治"漏掉 SSM/Mamba"的唯一办法**——占位符没填 = 没写完。

### 硬规则

- 报告**必须**覆盖到生成时点前 2 年的关键进展
- 2026 年生成 → 必须包含 2024-2026 进展
- 2024 年生成 → 必须包含 2022-2024 进展
- 快速演进的领域（AI、芯片、加密货币）→ 升级到前 3 年
- 特例：纯数学定理、30+ 年未变的协议等 → 标"低时效性领域"可豁免，但**必须**在事实预收集卡里说明理由

## 推断层使用规则（v2.4 新增）

**为什么需要**：v2.3 的预收集卡让 AI 写得"太安全"——字数降到 6500，缺乏灵感。**v2.4 允许"基于事实的合理推断"**填补这部分空间。

### 推断层的本质

**推断 ≠ 捏造**：
- 捏造 = 没有事实依据的断言（如"GPT-2 2018-11 发布"——AI 凭印象）
- 推断 = 基于事实的逻辑延伸（如"个人项目 5 年周期是正常的"——基于 573 commits + 5 年时间跨度）

### 推断的 3 条硬规则

1. **必须引用事实编号**——每条推断在 B 段预收集卡里**显式引用** A 段的事实编号（如"引用 A-#X, A-#Y"）
2. **不得包含具体数字/日期/人名**——那是 A 段的事。推断只能"定性"（"很短""约几年""少数"），不能"定量"
3. **占比 ≤ 50%**——事实主导，避免"看起来很有逻辑其实是猜测"

### 推断的常见形式（v2.3 vs v2.4 对比）

| v2.3（保守）| v2.4（推断层）|
|------------|--------------|
| 只写有来源的事实 | 事实 + 基于事实的合理推断 |
| 字数受限于事实卡 | 推断补足字数 |
| "灵感" = 0 | 推断是"灵感"的合法出口 |
| 全部标 [✓ 可靠] 或 [? 未知] | 加 [△ 推断] 区分 |

### 推断的边界（hard limit）

**禁止**的推断（即使有事实支撑）：
- ❌ "未来 2 年 X 将取代 Y"（预测）
- ❌ "X 是 Y 的根本原因"（强因果，无反事实证据）
- ❌ 包含具体数字/日期/人名的"推断"（绕过 A 段）
- ❌ 价值判断（"X 是好的/坏的"）

**允许**的推断：
- ✓ 趋势归纳（"个人项目 5 年周期是正常的"）
- ✓ 模式识别（"开源 + 闭源资产 = 反盗版模式"）
- ✓ 对比推断（"X 比 Y 在 Z 维度更强"——基于 A 段事实）
- ✓ 边界归纳（"X 不适合场景 Z，因为 A 段显示..."）

### 报告中的标注

```html
<!-- 事实层 -->
<span class="tag tag-reliable">[✓ 可靠][1]</span> 2017-03-08 创建
<span class="tag tag-unknown">[? 未知]</span> 多模式实现细节

<!-- 推断层 -->
<span class="tag tag-inference">[△ 推断]</span> 个人项目 5 年周期是正常的（引用 A-#X, A-#Y）

<!-- 模板对应 CSS（report.html） -->
.tag-inference { background: #d6eaf8; color: #1a5276; }
```

**这是 v2.4 的核心改变**：从"凭印象写"变成"**基于事实的灵感**"——既保持准确，又允许创造。

## 必启扩展维度

**对概念/技术模式，扩展维度不强制启用**——AI 可按需判断。

**如果 AI 决定启用任何扩展维度**，必须显式说明理由：

| 扩展维度 | 启用条件 |
|---------|---------|
| 优缺点 | 存在 2+ 个同类竞品 |
| 常见误解 | 公众对此有 ≥ 2 个常见错误理解 |
| 类比解释 | 普通读者需 ≥ 3 步才能理解 |
| 同类对比 | 存在 2+ 个可对比的同类 |

**禁用规则**：不允许"全不启用"——必须对每个扩展维度显式回答"启用/不启用 + 理由"。

## 禁用清单

### AI 味词（强制 grep 检测）

报告生成后**必须**用 grep 搜索以下词汇，**命中 ≤ 3 才允许交付**：

- `本质上` `说白了` `这意味着` `值得注意的是` `综上所述`
- `赋能` `抓手` `打造闭环` `助力`
- `激发` `促进` `推动` `保障`（近义套话）
- `在当今...的时代`（教科书开头）
- `首先...其次...最后`（套话结构）
- `非常` `极其` `相当` `想象空间巨大`（空洞形容词）

### 写作风格

- 教科书开头（"在当今 XXX 快速发展的时代"）
- 套话堆砌（"AI Agent 是大势所趋")
- 居高临下定论（避免"必然是" "一定是"）
- 抽象泛化（"具有重大意义""发挥重要作用"）

## 写作原则

| 原则 | 说明 |
|------|------|
| **叙事驱动** | 用故事线串信息，不是年表罗列 |
| **具体细节** | 用数据替代"快速增长"等概括 |
| **层层剥开** | 现象 → 表面解释 → 深层追问 → 核心洞察 |
| **回环呼应** | 开头埋的钩子，结尾 callback |
| **承认主观** | 用"基于以上分析可推断"而非居高临下的定论 |

## 一句话总结

**概念/技术模式 = "事前提取（双层卡）+ 事中写作（事实+推断）+ 事后检查（占位符+推断审计）"。**

- **事前提取**：A 段事实地基（必填 + [N]）+ B 段推断上层（必填 + 引用 A 段）——**先于任何写作**
- **事中写作**：用 A 段写事实（[✓ 可靠][N]）+ 用 B 段写推断（[△ 推断]）
- **事后检查**：质检报告检查"预收集齐全 + 占位符填满 + 推断引用 A 段 + 推断 ≤ 50%"

**v2.4 核心变化**：从"单层预收集"升级为"双层预收集"——**事实严格 + 推断宽松**。既保持准确（v2.3 优势），又恢复灵感（v2.0 优势）。

**v2.3 的核心变化**：从"事后补规则"改成"事前提取"——AI 不能再"凭印象写，写完假装自检"。
